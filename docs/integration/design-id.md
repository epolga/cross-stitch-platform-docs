# Contract: DesignID

## 1. Contract Name

**DesignID** — cross-repo identifier for individual designs within an album.

`DesignID` is a globally unique, monotonically increasing positive integer that identifies one cross-stitch design (one PDF "kit"). It is generated by the Uploader (WPF, C#) and consumed by the cross-stitch web app (Next.js, TypeScript) as the addressable key for design pages, S3 photo/PDF assets, the DynamoDB `DesignsByID-index` GSI, and the design API.

- Producer-side declaration: `d:/ann/Git/Uploader/Uploader/PatternInfo.cs:90-94` — `public int DesignID { get; set; }` (XML doc: "Unique design ID in your system.")
- Consumer-side declaration: `d:/ann/Git/cross-stitch/src/app/types/design.ts:4` — `DesignID: number;`

---

## 2. Purpose

`DesignID` exists so that every uploaded design has one stable handle that can:

1. Address the design's row in DynamoDB via the `DesignsByID-index` GSI (`d:/ann/Git/Uploader/Uploader/MainWindow.xaml.cs:838`, `d:/ann/Git/cross-stitch/src/lib/data-access.ts:581-588`).
2. Build the canonical S3 photo prefix (`photos/{AlbumID}/{DesignID}/`) and PDF object keys (`pdfs/{AlbumID}/Stitch{DesignID}_Kit.pdf`).
3. Drive the consumer-facing design page route `/designs/[designId]` (`d:/ann/Git/cross-stitch/src/app/designs/[designId]/page.tsx:17-19`) and its backing JSON API (`d:/ann/Git/cross-stitch/src/app/api/designs/[designId]/route.ts:3-22`).
4. Correlate analytics, Pinterest pins, and download-count updates (`d:/ann/Git/cross-stitch/src/app/components/DownloadPdfLink.tsx:66-69`).

The companion concept `AlbumID` identifies a *collection* of designs; `DesignID` identifies a *single* design within (or across) those collections. See §8 for the AlbumID cross-reference.

## 3. Scope

**In-scope**

- The integer value flowing between the Uploader (writer) and cross-stitch (reader) through DynamoDB, S3, and HTTP URLs.
- Generation, allocation, and uniqueness guarantees in the Uploader (`GetNextDesignIdAsync`, `MainWindow.xaml.cs:833-857`).
- All places it appears as a path segment, attribute, route param, or query value.

**Out-of-scope**

- The internal PDF "Stitch number" suffix (`_1_Kit.pdf`, `_3_Kit.pdf`, `_5_Kit.pdf`) — that is a PDF-format concern, covered by the PDF path contract; this contract only states that DesignID appears in the filename pattern.
- Per-user "favorited design" lists, which use `DesignID` as a foreign key but are governed by the `CrossStitchUsers` contract.
- Pinterest pin IDs (the `PinID` attribute on the same DynamoDB row) — see the Pinterest pin-ID contract.

## 4. Data Formats

### 4.1 Logical type

`DesignID` is a **raw decimal integer**, transported as:

| Surface | Encoding | Example |
|---|---|---|
| C# field | `int` | `1234` |
| DynamoDB attribute `DesignID` | Number (`N`) | `"1234"` |
| TypeScript field on `Design` | `number` | `1234` |
| URL path segment (`/designs/{designId}`) | bare decimal string | `1234` |
| S3 path segment (`photos/{AlbumID}/{DesignID}/`) | bare decimal string | `4567/1234/` |
| Internal CSV/SCC key (`charts/{designId:D5}_…`) | 5-digit zero-padded — *Uploader internal only, not part of this contract* | `01234` |

**No zero-padded form is part of the cross-repo contract.** The only place padding is observed is the Uploader-internal SCC chart key at `d:/ann/Git/Uploader/Uploader/MainWindow.xaml.cs:932-933` (`designId.ToString("D5")`), which the cross-stitch app never reads.

Default sentinel = `-1` in the Uploader before assignment (`d:/ann/Git/Uploader/Uploader/PatternInfo.cs:27` — `private int _designId = -1;`). Valid DesignIDs are always `≥ 1` (allocation starts at `1` per `MainWindow.xaml.cs:856` — `return 1;`).

### 4.2 Allocation algorithm (next-id sweep)

```csharp
// d:/ann/Git/Uploader/Uploader/MainWindow.xaml.cs:833-857
TableName = "CrossStitchItems",
IndexName = "DesignsByID-index",
KeyConditionExpression = "EntityType = :et",   // ":et" = "DESIGN"
ScanIndexForward = false,
Limit = 1,
ProjectionExpression = "DesignID"
// returns int.Parse(response.Items[0]["DesignID"].N) + 1;  // or 1 if empty
```

This means DesignID is **assigned at upload time as `max(DesignID where EntityType=DESIGN) + 1`** and is re-computed (`MainWindow.xaml.cs:642-643`) immediately before the S3/DynamoDB write to minimise the race window. There is no transactional reservation — see §9.

### 4.3 Relationship to `NPage` (compound DDB key)

The DynamoDB primary key of the `CrossStitchItems` table is `(ID, NPage)` — confirmed by:

- Uploader query usage: `KeyConditionExpression = "ID = :id"` with `ScanIndexForward = false` and `ProjectionExpression = "NPage"` (`d:/ann/Git/Uploader/Uploader/MainWindow.xaml.cs:809-818`).
- Uploader PutItem payload: `ID` and `NPage` are the first two attributes set (`MainWindow.xaml.cs:1067-1068`).
- Cross-stitch update path: `Key: { ID: { S: id }, NPage: { S: nPage } }` (`d:/ann/Git/cross-stitch/src/lib/data-access.ts:628-631`).

A design row's primary key is therefore `ID = "ALB#0001"` (or similar 4-digit-padded album partition, `MainWindow.xaml.cs:107` — `AlbumPartitionKey = $"ALB#{albumId:D4}";`) and `NPage = "00001"` (5-digit zero-padded string, `MainWindow.xaml.cs:830` — `(maxNPage + 1).ToString("D5")`).

`DesignID` is **NOT** part of the primary key. It is a stand-alone integer attribute and the partition key of a **GSI** named `DesignsByID-index` (`MainWindow.xaml.cs:838`, `data-access.ts:581`). The relationship is:

- `(ID, NPage)` ⇒ uniquely identifies a design row within an album.
- `DesignID` ⇒ uniquely identifies the same row across all albums, via the GSI.

Both keys must round-trip to the same row; the cross-stitch cache stores both (`data-access.ts:178-180`) so it can convert `DesignID → (ID, NPage)` for updates.

## 5. API Endpoints / Interfaces

There is no formal HTTP service for `DesignID` itself; it is a value that flows through five interface surfaces.

### 5.1 Web URL — design detail page

- Route: `/designs/[designId]` (`d:/ann/Git/cross-stitch/src/app/designs/[designId]/page.tsx:17-19`)
- Backing API: `GET /api/designs/{designId}` (`d:/ann/Git/cross-stitch/src/app/api/designs/[designId]/route.ts:3-22`) — returns the `Design` JSON object.
- Update API: `POST /api/designs/{designId}` increments `NDownloaded` (`route.ts:32-49`).

### 5.2 Web URL — pretty slug (NPage-based, not DesignID-based)

- Pattern: `/{Caption}-{AlbumID}-{NPage-1}-Free-Design.aspx`
- Producer: `PatternLinkHelper.BuildPatternUrl` (`d:/ann/Git/Uploader/Uploader/Helpers/PatternLinkHelper.cs:62-71`).
- Consumer: `CreateDesignUrl` (`d:/ann/Git/cross-stitch/src/lib/url-helper.ts:54-59`).

**Note:** this slug does **not** embed `DesignID` — it uses `NPage-1` instead. The cross-stitch resolver therefore needs both the `Caption`/`AlbumID`/`NPage` triple and the `DesignID` (cached per `data-access.ts:178-180`) to navigate between the two surfaces.

### 5.3 DynamoDB attribute & GSI

- Table: `CrossStitchItems` (`MainWindow.xaml.cs:809, 1085`).
- Attribute name: `DesignID` (number); written at `MainWindow.xaml.cs:1072`, read at `data-access.ts:142`.
- GSI name: `DesignsByID-index` (`MainWindow.xaml.cs:838`, `data-access.ts:581`) — partition key `EntityType`, sort key `DesignID`.
- Lookup pattern: `EntityType = "DESIGN" AND DesignID = :designId` to translate `DesignID → (ID, NPage)` (`data-access.ts:582-588`).

### 5.4 S3 object keys

Photo prefix (Uploader writer at `MainWindow.xaml.cs:2229-2232`, reader at `data-access.ts:153-154`):

```
photos/{AlbumID}/{DesignID}/{filename}.jpg     # e.g. photos/42/5366/4.jpg
```

PDF "main" kit (Uploader at `MainWindow.xaml.cs:957`, reader at `data-access.ts:367`):

```
pdfs/{AlbumID}/Stitch{DesignID}_Kit.pdf        # e.g. pdfs/42/Stitch5366_Kit.pdf
```

PDF per-variant kit (Uploader at `MainWindow.xaml.cs:958-961`, reader at `DownloadPdfLink.tsx:74`):

```
pdfs/{AlbumID}/{DesignID}/Stitch{DesignID}_{N}_Kit.pdf
# N ∈ {1, 3, 5} on the producer side
# e.g. pdfs/42/5366/Stitch5366_3_Kit.pdf
```

Public read URLs use CloudFront: `https://d2o1uvvg91z7o4.cloudfront.net/<key>` (`data-access.ts:154, 367`).

### 5.5 PDF filename patterns (recap)

| Pattern | Where written | Where read |
|---|---|---|
| `Stitch{DesignID}_Kit.pdf` (legacy "main") | `MainWindow.xaml.cs:957` | `data-access.ts:367` |
| `Stitch{DesignID}_{N}_Kit.pdf` (variants `_1_/_3_/_5_`) | `MainWindow.xaml.cs:959-961` | `DownloadPdfLink.tsx:74` |

## 6. Versioning

**Unversioned (implicit).** Neither repo embeds a contract version number alongside `DesignID`; the integer value itself is the only identifier.

Recommended: when the storage format changes (e.g., switching to a UUID or to a strictly partitioned ID space per album), introduce a `DesignIDv2` attribute or migrate via a new GSI named `DesignsByID-v2-index` rather than reusing the existing one — backfill, dual-write, then cut over consumers.

## 8. Dependencies

`DesignID` depends on, and is depended on by, the following contracts:

1. **AlbumID** — an album owns multiple designs. AlbumID is the partition prefix for both DDB (`ID = "ALB#{AlbumID:D4}"`, `MainWindow.xaml.cs:107`) and S3 paths (`photos/{AlbumID}/{DesignID}/…`, `pdfs/{AlbumID}/…`). The S3-key contracts in §5.4 are *joint* AlbumID+DesignID contracts.
2. **DynamoDB schema (`CrossStitchItems`)** — defines the `(ID, NPage)` primary key and the `DesignsByID-index` GSI on `(EntityType, DesignID)`. Without that GSI, the cross-stitch app cannot resolve a `DesignID` to a row (`data-access.ts:581-588`).
3. **S3 key conventions** — both photo and PDF prefixes are duplicated string templates across repos (no shared constant); see `PatternLinkHelper.BuildImageUrl` (`PatternLinkHelper.cs:73-76`) vs. `data-access.ts:153-154`.
4. **PDF kit structure** — the `Stitch{DesignID}_{N}_Kit.pdf` and `Stitch{DesignID}_Kit.pdf` naming is a *PDF-product* convention; this contract only fixes that `DesignID` appears in those filenames.
5. **Pinterest pin record** — the same DynamoDB row that owns a `DesignID` also stores `PinID`; see the Pinterest pin-ID contract.
6. **`NGlobalPage`** — an independent global counter computed at upload time (`MainWindow.xaml.cs:637-638`) and used only for chronological ordering in the cross-stitch home feed (`data-access.ts:422`). It is not equivalent to `DesignID`.

## 9. Error Handling

### 9.1 URL parsing (`/designs/[designId]`)

- `parseInt(designId, 10)` is the only validation; non-numeric values return `400 Invalid designId` (`d:/ann/Git/cross-stitch/src/app/api/designs/[designId]/route.ts:8-11`).
- The page wrapper at `d:/ann/Git/cross-stitch/src/app/designs/[designId]/page.tsx:158-173` catches fetch failures and renders a "Design Not Found" error block, including the raw error message in the UI — observed behaviour, no PII risk for an integer ID, but the page still logs `console.error('Error fetching design:', error)`.

### 9.2 DynamoDB miss

- Cache-first lookup at `data-access.ts:371-385`: if `designCache.get(designId)` returns `undefined`, the function logs `No design found for DesignID ${designId} in cache` and returns `undefined`. The route handler then emits `404 Design not found` (`route.ts:17-19`).
- Direct GSI fallback for download-count updates (`data-access.ts:573-604`): if the GSI returns zero items, `fetchDesignKeyFromDb` returns `null` and the caller throws `Unable to locate DynamoDB key info for design ${designId}` (`data-access.ts:619-621`). This is a hard failure — the increment is dropped.

### 9.3 Allocation race / orphan handling

- `GetNextDesignIdAsync` is **not transactional**: two concurrent uploads can compute the same "next" `DesignID` (`MainWindow.xaml.cs:833-857`). The flow at `MainWindow.xaml.cs:642-648` *recomputes* `DesignID` right before the writes to shrink the window, but does not eliminate it.
- Writes are not atomic across S3 + DDB. If S3 photo/PDF uploads succeed but the DynamoDB `PutItem` at `MainWindow.xaml.cs:1083-1089` fails (or Pinterest pinning fails first at `MainWindow.xaml.cs:656-659`), the S3 objects under `photos/{AlbumID}/{DesignID}/` and `pdfs/{AlbumID}/Stitch{DesignID}_Kit.pdf` remain as **orphans** — no compensating delete exists in the code path.
- The reader has a known orphan-PDF guard: `MissingDesignPdfs.txt` enumerates `DesignID`s whose PDFs are missing from S3 (`d:/ann/Git/cross-stitch/src/app/designs/[designId]/page.tsx:47-66, 175-176`).
- One hard-coded skip on the Uploader cleanup side: `if (design.DesignId == 5366) … missing.Add(...)` (`MainWindow.xaml.cs:3635`) — a known broken DesignID treated as missing rather than re-uploaded.

### 9.4 Attribute-shape errors

- `ParseRequiredIntAttribute(item, "DesignID")` throws `InvalidOperationException("Latest design attribute 'DesignID' is missing or invalid in DynamoDB.")` (`MainWindow.xaml.cs:880, 900-912`) if the DDB row lacks `DesignID` or it is non-numeric — fatal for the newsletter flow.

## 10. Security & Compliance

- `DesignID` is **not** sensitive — it is a sequential integer surfaced in public URLs (`/designs/123`), public S3/CloudFront paths (`https://d2o1uvvg91z7o4.cloudfront.net/photos/42/5366/4.jpg`, `data-access.ts:154`), and SEO metadata (`page.tsx:93-96`).
- Sequential allocation does leak **total design count** to anyone who can read public URLs — acceptable for a free-pattern catalog.
- No secrets-on-disk concern is introduced by this contract; the only writer-side credential needed to allocate a `DesignID` is the AWS access key used by the Uploader to query `CrossStitchItems` (configured via `App.config` / IAM and out of scope here).

## 11. Testing & Validation

Concrete checks to verify the contract holds:

1. **Grep for path patterns matching `Stitch{DesignID}_Kit.pdf`** across both repos to catch new producers/consumers that bypass the shared template:
   ```
   rg -n "Stitch\{?[^}]*[Dd]esign[Ii][Dd]?\}?_(?:\d+_)?Kit\.pdf" \
     d:/ann/Git/Uploader d:/ann/Git/cross-stitch
   ```
2. **Grep for photo prefix usage** to confirm no new code path constructs `photos/{AlbumID}/{DesignID}/`:
   ```
   rg -n "photos/\$?\{?[Aa]lbum[Ii][Dd]\}?/\$?\{?[Dd]esign[Ii][Dd]\}?/" \
     d:/ann/Git/Uploader d:/ann/Git/cross-stitch
   ```
3. **DDB integrity probe** — pick a random `DesignID`, query the `DesignsByID-index` GSI on the consumer side (`data-access.ts:582-588`), then `Get` the row by `(ID, NPage)` and confirm `DesignID` on the row matches.
4. **Allocation race probe** — run two concurrent simulated uploads against a dev DDB table and assert no duplicate `DesignID` survives; today this would fail (see §9.3) and should be tracked.
5. **Orphan scan** — list `s3://cross-stitch-designs/photos/*/*/` and `s3://cross-stitch-designs/pdfs/*/Stitch*_Kit.pdf` prefixes and diff against `DesignID`s present in DDB; any S3-only `DesignID` is an orphan.
6. **Type-shape probe** — assert `parseInt(designId, 10)` succeeds for every `DesignID` returned by `/api/designs` (regression guard for the route at `route.ts:8-11`).
